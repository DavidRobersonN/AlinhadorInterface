1. Ideia geral da arquitetura

A ideia principal é transformar o Raspberry Pi no cérebro da máquina e deixar o Arduino como uma placa de entrada e saída física.

Arduino

  • Lê sensores
  • Aciona motor
  • Aciona cilindro
  • Executa comandos rápidos
  • Responde ao Raspberry

Raspberry + Django

  • Interpreta leituras
  • Aplica regras de decisão
  • Decide apertar ou afrouxar
  • Coordena o processo
  • Envia status para a interface
React ⇄ WebSocket ⇄ Django no Raspberry ⇄ Serial USB ⇄ Arduino ⇄ Sensores / Motores / Cilindros

Leitura mental correta

Em vez de colocar toda a inteligência dentro do microcontrolador, você pode deixar o Arduino responsável pelo mundo físico e o Django responsável pela lógica de decisão.

Resumo simples: Arduino = camada física. Django = camada de decisão.

2. O que fica no Arduino e o que fica no Django

Bom deixar no Arduino

  • Leitura rápida de pinos
  • Debounce de botão
  • Controle fino de pulso de motor
  • Acionamento instantâneo de relé
  • Tempos críticos de hardware
  • Rotinas físicas imediatas

Bom deixar no Django

  • Interpretação das leituras
  • Comparação com tolerâncias
  • Estratégia de alinhamento
  • Decisões de alto nível
  • Fluxo do processo
  • Histórico, interface e logs
Cuidado importante: o Django não deve gerar pulsos de stepper com precisão de microsegundos. Esse tipo de controle fino continua sendo melhor no Arduino.

3. Quando faz sentido deixar a lógica no Django

Isso faz sentido quando a lógica começa a ficar mais inteligente, por exemplo:

  • Se o desvio lateral for maior que X e a tensão estiver abaixo de Y, então aperta 1/4 de volta.
  • Se corrigiu demais, volta 1/8.
  • Se a leitura vertical estiver fora do limite, usa outra estratégia.
  • Se a tensão da região estiver muito alta, para de apertar.
  • Se o raio atual não for o certo, gira a roda até o próximo.

Esse tipo de combinação de regras fica muito mais confortável de escrever em Python do que dentro do código do Arduino.

4. Vantagens dessa arquitetura

  1. Código mais organizado: regras complexas ficam mais limpas em Python.
  2. Mais fácil de crescer: você pode adicionar banco de dados, calibração, histórico, relatórios e logs.
  3. Mais fácil de alterar regras: mudar uma estratégia de correção no Django costuma ser bem mais simples.
  4. Melhor integração com interface: o Django já conversa com o React e consegue mostrar tudo em tempo real.
Na prática: o Django pode mostrar na tela o valor dos sensores, a decisão tomada, o estado do motor, a etapa atual da máquina e um histórico do que foi feito.

5. Fluxo prático do sistema

Vamos imaginar sua máquina com estes itens:

  • Sensor lateral
  • Sensor vertical
  • Sensor de tensão
  • Motor para girar nipple

Fluxo de alto nível

  1. Arduino lê os sensores.
  2. Arduino envia as leituras para o Raspberry.
  3. Django interpreta as leituras.
  4. Django decide o que fazer.
  5. Django envia comando para o Arduino.
  6. Arduino executa o movimento.
  7. Arduino devolve confirmação.
  8. Django pede nova leitura.
  9. O ciclo se repete até ficar dentro da tolerância.
LER SENSORES → AVALIAR → DECIDIR → EXECUTAR → MEDIR DE NOVO

6. Exemplo de dados enviados pelo Arduino

O Arduino pode mandar os dados pela serial em formato JSON.

sensores.json
{
  "type": "sensor_data",
  "raio": 12,
  "lateral": 1.8,
  "vertical": 0.6,
  "tensao": 92
}

Ou também pode usar um formato de string simples pela serial:

serial.txt
SENSORES:12,1.8,0.6,92

Nesse exemplo, significaria: raio 12, lateral 1.8, vertical 0.6 e tensão 92.

7. Exemplo de decisão no Django

Aqui o Raspberry recebe os valores e aplica uma regra. Um exemplo simples seria:

avaliacao.py
def avaliar_alinhamento(lateral, vertical, tensao):
    if lateral > 1.5 and tensao < 100:
        return {
            "acao": "apertar",
            "passos": 100,
            "motivo": "desvio lateral alto com tensão segura"
        }

    if lateral < -1.5 and tensao < 100:
        return {
            "acao": "afrouxar",
            "passos": 100,
            "motivo": "desvio lateral para o lado oposto"
        }

    if abs(vertical) > 1.0:
        return {
            "acao": "corrigir_vertical",
            "passos": 50,
            "motivo": "desvio vertical acima do limite"
        }

    return {
        "acao": "nenhuma",
        "passos": 0,
        "motivo": "roda dentro da tolerância"
    }

O que essa função faz

  • Analisa desvio lateral.
  • Analisa desvio vertical.
  • Considera o limite de tensão.
  • Retorna uma decisão já pronta para ser executada.
Vantagem: essa regra pode crescer com muito mais facilidade no Django do que no Arduino.

8. Exemplo de uso dessa lógica

Suponha que o Django receba esta leitura:

dados.py
dados = {
    "lateral": 1.8,
    "vertical": 0.4,
    "tensao": 90,
}

Ele pode chamar a função assim:

uso.py
decisao = avaliar_alinhamento(
    lateral=dados["lateral"],
    vertical=dados["vertical"],
    tensao=dados["tensao"]
)

Resultado esperado:

resultado.json
{
    "acao": "apertar",
    "passos": 100,
    "motivo": "desvio lateral alto com tensão segura"
}

Depois disso, o Django poderia enviar este comando para o Arduino:

serial_command.py
send_command_to_arduino("APERTAR:100")

9. Exemplo mais próximo da sua máquina

machine_controller.py
class ControladorAlinhamento:
    def __init__(self):
        self.tolerancia_lateral = 0.5
        self.tolerancia_vertical = 0.5
        self.tensao_maxima = 120
        self.tensao_minima = 80

    def decidir(self, leitura):
        lateral = leitura["lateral"]
        vertical = leitura["vertical"]
        tensao = leitura["tensao"]

        if abs(lateral) <= self.tolerancia_lateral and abs(vertical) <= self.tolerancia_vertical:
            return {"comando": "PARAR", "motivo": "Roda alinhada"}

        if lateral > self.tolerancia_lateral:
            if tensao < self.tensao_maxima:
                return {"comando": "APERTAR", "passos": 100, "motivo": "Corrigir lateral"}
            return {"comando": "PARAR", "motivo": "Tensão alta demais"}

        if lateral < -self.tolerancia_lateral:
            if tensao > self.tensao_minima:
                return {"comando": "AFROUXAR", "passos": 100, "motivo": "Corrigir lateral oposta"}
            return {"comando": "PARAR", "motivo": "Tensão baixa demais"}

        return {"comando": "PARAR", "motivo": "Sem ação"}

Leitura desse código

  • Define tolerâncias.
  • Compara a leitura recebida com essas tolerâncias.
  • Decide entre apertar, afrouxar ou parar.
  • Também protege contra tensão muito alta ou muito baixa.

11. Exemplo de sequência automática

Imagine o seguinte ciclo automático de correção:

  1. Django manda: LER_SENSORES
  2. Arduino responde com os valores atuais.
  3. Django interpreta os dados.
  4. Django decide uma ação.
  5. Arduino executa a ação.
  6. Django pede nova leitura.
  7. O processo repete até ficar dentro da tolerância.
ciclo_exemplo.txt
# Etapa 1
LER_SENSORES

# Etapa 2
{"type":"sensor_data","raio":8,"lateral":2.1,"vertical":0.3,"tensao":88}

# Etapa 3 - decisão do Django
{"comando": "APERTAR", "passos": 100}

# Etapa 4
APERTAR:100

# Etapa 5
APERTAR_DONE

# Etapa 6
LER_SENSORES

# Etapa 7
{"type":"sensor_data","raio":8,"lateral":0.9,"vertical":0.3,"tensao":95}

# Etapa 8 - nova decisão
APERTAR:50

Esse é o tipo de comportamento que transforma o Django em um controle supervisor da máquina.

Ele não precisa gerar cada pulso do motor. Ele só precisa decidir o que fazer, quando fazer, quanto fazer e quando parar.

13. Exemplo simples de machine_controller.py

machine_controller.py
class MachineController:
    def processar_leitura(self, leitura):
        lateral = leitura["lateral"]
        vertical = leitura["vertical"]
        tensao = leitura["tensao"]

        if abs(lateral) > 1.5:
            if tensao < 110:
                return {"serial_command": "APERTAR:100", "status": "Corrigindo lateral"}
            else:
                return {"serial_command": "PARAR", "status": "Tensão acima do limite"}

        if abs(vertical) > 1.0:
            return {"serial_command": "CORRIGIR_VERTICAL:50", "status": "Corrigindo vertical"}

        return {"serial_command": "PARAR", "status": "Dentro da tolerância"}

Quando chegar uma leitura do Arduino, o Django pode fazer algo assim:

uso_controller.py
controller = MachineController()
decisao = controller.processar_leitura(leitura)
send_command_to_arduino(decisao["serial_command"])

14. Conclusão prática

Para a sua máquina, a arquitetura mais interessante é esta:

No Arduino

  • Sensores
  • Motores
  • Cilindros
  • Rotinas rápidas
  • Comandos básicos

No Raspberry com Django

  • Lógica de alinhamento
  • Estratégia de correção
  • Comparação de leituras
  • Sequência do processo
  • Histórico e interface
Regra de ouro: decisão inteligente no Django; controle físico fino no Arduino.

Essa separação deixa o projeto mais profissional, mais organizado e muito mais fácil de crescer no futuro.
