1. Como o seu programa está organizado

Seu projeto está dividido em 3 partes principais. Cada uma tem uma responsabilidade bem específica.

A. React

É a interface.

  • Botão na tela
  • Texto “conectado”
  • Texto “LED ligado/desligado”

O React não acende o LED diretamente. Ele apenas envia uma mensagem.

B. Django + Channels

É o meio de campo.

  • Recebe mensagem do React pelo WebSocket
  • Manda comando para o Arduino pela serial USB
  • Recebe resposta do Arduino
  • Repassa essa resposta ao React

Na prática, o Django funciona como uma ponte.

C. Arduino

É quem controla o hardware real.

  • Lê o botão físico
  • Acende e apaga o LED
  • Recebe comandos do computador
  • Avisa de volta qual é o estado atual
React → WebSocket → Django → Serial → Arduino → Hardware

2. Caminho completo: da tela até o LED acender

Agora vamos seguir o sinal, etapa por etapa, até o LED realmente acender.

Etapa 1 — você clica no botão da interface

No React, existe um botão assim:

App.tsx
<button onClick={() => sendCommand('ON')}>Ligar</button>

Quando você clica, o React executa a função sendCommand('ON').

Etapa 2 — a função sendCommand() roda

App.tsx
const sendCommand = (command: string) => {
  if (socketRef.current && socketRef.current.readyState === WebSocket.OPEN) {
    socketRef.current.send(JSON.stringify({ command }))
  }
}
O que acontece aqui:
  • socketRef.current é a conexão WebSocket aberta com o Django.
  • readyState === WebSocket.OPEN verifica se a conexão está aberta.
  • send(JSON.stringify({ command })) envia a mensagem ao backend.

O React manda algo assim:

mensagem.json
{"command":"ON"}

Etapa 3 — o Django recebe a mensagem no consumer

consumers.py
async def receive(self, text_data):
    data = json.loads(text_data)
    command = data.get("command")

    if command == "ON":
        send_command_to_arduino("ON")
    elif command == "OFF":
        send_command_to_arduino("OFF")
    elif command == "TOGGLE":
        send_command_to_arduino("TOGGLE")

text_data chega como texto puro, por exemplo {"command":"ON"}. Depois o json.loads transforma isso em dicionário Python e o Django extrai o valor do comando.

Etapa 4 — o Django escreve na serial USB

serial_manager.py
def send_command_to_arduino(command):
    global arduino
    if arduino and arduino.is_open:
        arduino.write((command + '\n').encode())
Ponto importante: o \n é a quebra de linha. Ela é importante porque no Arduino a leitura está sendo feita com Serial.readStringUntil('\n').

Na prática, o Django manda para o Arduino:

serial.txt
ON

3. Continuação do fluxo até o LED acender

Etapa 5 — o Arduino recebe o comando

Arduino
void handleSerial() {
  if (Serial.available() > 0) {
    String command = Serial.readStringUntil('\n');
    command.trim();

    if (command == "ON") {
      ledState = true;
      updateLed();
      sendLedState();
    }
    else if (command == "OFF") {
      ledState = false;
      updateLed();
      sendLedState();
    }
    else if (command == "TOGGLE") {
      ledState = !ledState;
      updateLed();
      sendLedState();
    }
  }
}

O Arduino verifica se chegou dado pela serial, lê até a quebra de linha, remove espaços extras e compara o comando recebido.

Etapa 6 — o Arduino acende o LED de verdade

Depois ele chama:

Arduino
updateLed();

que executa:

Arduino
void updateLed() {
  digitalWrite(ledPin, ledState ? HIGH : LOW);
}

Se ledState for true, então o Arduino faz:

Arduino
digitalWrite(ledPin, HIGH);
É aqui que o hardware realmente muda e o LED acende.

Etapa 7 — o Arduino avisa o computador do novo estado

Logo depois ele chama:

Arduino
sendLedState();

que faz:

Arduino
void sendLedState() {
  if (ledState) {
    Serial.println("LED_ON");
  } else {
    Serial.println("LED_OFF");
  }
}

Isso é muito importante, porque a interface não deve assumir que o LED acendeu. Ela deve receber a confirmação real do Arduino.

Etapa 8 — o Django escuta a serial o tempo todo

serial_manager.py
def listen_to_arduino():
    global arduino
    channel_layer = get_channel_layer()

    while True:
        try:
            if arduino and arduino.is_open and arduino.in_waiting > 0:
                line = arduino.readline().decode().strip()

                if line == 'LED_ON':
                    async_to_sync(channel_layer.group_send)(
                        'led_group',
                        {
                            'type': 'led_status',
                            'state': 'ON'
                        }
                    )

                elif line == 'LED_OFF':
                    async_to_sync(channel_layer.group_send)(
                        'led_group',
                        {
                            'type': 'led_status',
                            'state': 'OFF'
                        }
                    )

Essa thread funciona como um vigia da serial: ela fica olhando continuamente se o Arduino mandou alguma coisa.

4. Final do fluxo até a interface atualizar

Etapa 9 — o consumer envia isso para o React

consumers.py
async def led_status(self, event):
    await self.send(text_data=json.dumps({
        "type": "led_status",
        "state": event["state"]
    }))

Então o Django manda para o frontend uma mensagem assim:

mensagem_saida.json
{
  "type": "led_status",
  "state": "ON"
}

Etapa 10 — o React recebe a mensagem

App.tsx
socket.onmessage = (event: MessageEvent) => {
  const data = JSON.parse(event.data)

  if (data.type === 'led_status') {
    setLedState(data.state === 'ON' ? 'Ligado' : 'Desligado')
  }
}

Quando setLedState(...) é executado, o React atualiza o estado do componente e renderiza a tela novamente. Aí aparece para o usuário:

  • Ligado
  • ou Desligado
Tela → React → WebSocket → Django → Serial → Arduino → Serial → Django → WebSocket → React → Tela

5. O caminho contrário: botão físico no Arduino

Agora vamos ver o fluxo no sentido inverso.

Você aperta o botão físico

O Arduino roda continuamente o loop principal:

Arduino
void loop() {
  handleButton();
  handleSerial();
}

Então, a cada repetição, ele verifica o botão. Quando o botão físico é apertado, a função handleButton():

  • detecta o clique
  • troca ledState
  • chama updateLed()
  • chama sendLedState()

O resultado é:

  1. o LED muda fisicamente
  2. o Arduino envia LED_ON ou LED_OFF
  3. o Django recebe pela serial
  4. o Django manda pelo WebSocket
  5. o React atualiza a tela
Resultado final: o botão físico também atualiza a interface web.

6. Papel de cada arquivo no projeto

App.tsx

  • Abrir WebSocket
  • Enviar comandos
  • Receber estados
  • Atualizar a tela

consumers.py

  • Receber mensagem do React
  • Chamar função que envia ao Arduino
  • Repassar mensagens ao React

serial_manager.py

  • Abrir a serial
  • Mandar comando para Arduino
  • Ler o que vem do Arduino
  • Repassar isso para o Channels

routing.py e asgi.py

  • routing.py define qual URL usa qual consumer
  • asgi.py habilita o Django para aceitar WebSocket

Exemplo do routing.py:

routing.py
re_path(r'ws/led/$', LedConsumer.as_asgi())

Isso significa: quando alguém conectar em /ws/led/, use o LedConsumer.

7. Por que usamos WebSocket e não fetch?

Se você usasse fetch, o React teria que ficar perguntando o tempo todo:

  • “e agora?”
  • “e agora?”
  • “e agora?”

Isso se chama polling.

Com WebSocket, a conexão fica aberta. Então o backend avisa o React no exato momento em que algo muda.

Isso é ideal para:

  • sensores
  • LEDs
  • motores
  • estados de máquina
  • monitoramento em tempo real

8. Exemplo: adicionar um botão para ligar um motor

Agora imagine que você quer adicionar um motor à lógica do projeto.

6.1 No React

Você adicionaria botões assim:

App.tsx
<button onClick={() => sendCommand('MOTOR_ON')}>Ligar motor</button>
<button onClick={() => sendCommand('MOTOR_OFF')}>Desligar motor</button>

Então o React passa a mandar mensagens como estas:

comando_motor.json
{"command":"MOTOR_ON"}
{"command":"MOTOR_OFF"}

6.2 No Django — consumers.py

consumers.py
async def receive(self, text_data):
    data = json.loads(text_data)
    command = data.get("command")

    if command == "ON":
        send_command_to_arduino("ON")
    elif command == "OFF":
        send_command_to_arduino("OFF")
    elif command == "TOGGLE":
        send_command_to_arduino("TOGGLE")
    elif command == "MOTOR_ON":
        send_command_to_arduino("MOTOR_ON")
    elif command == "MOTOR_OFF":
        send_command_to_arduino("MOTOR_OFF")

O Django continua sendo apenas a ponte. Ele não precisa saber como o motor funciona por dentro.

10. O mais importante de entender

Seu sistema funciona como uma cadeia.

Quando vem da tela:

React → WebSocket → Django → Serial → Arduino → hardware

Quando vem do botão físico ou sensor:

Arduino → Serial → Django → WebSocket → React

11. Mentalidade certa para crescer o projeto

Sempre pense em 3 camadas:

Camada 1 — Interface

Botões, textos, telas.

React

Camada 2 — Comunicação

WebSocket e serial.

Django

Camada 3 — Hardware

LED, motor, sensor, cilindro, relé.

Arduino

Se você organizar assim, fica muito mais fácil crescer para:

  • motor de passo
  • sensores magnéticos
  • contador de raio
  • pistões pneumáticos
  • câmera
  • painel de status da máquina

12. Resumo bem simples

Quando você clica no botão da tela:

  1. React envia comando pelo WebSocket
  2. Django recebe
  3. Django manda pela serial
  4. Arduino recebe
  5. Arduino acende o LED
  6. Arduino confirma o novo estado
  7. Django lê essa confirmação
  8. Django manda para o React
  9. React atualiza a tela

Quando aperta o botão físico:

  1. Arduino detecta
  2. Arduino muda o LED
  3. Arduino envia o novo estado
  4. Django recebe
  5. Django manda ao React
  6. React atualiza a tela
Ideia central: React mostra, Django faz a ponte, Arduino executa.
